# Task Elapsed Time Design

## Goal

Record and display the elapsed time between a student's explicit start action and the final successful homework submission. The metric is named "本次任务耗时" and must not be described as focused or active study time.

## Scope

- Use server timestamps as the source of truth.
- Keep one unfinished study session per daily task.
- Recover that unfinished session after the mini-program is hidden, restarted, or re-enters the timer page.
- End the linked session when the homework submission is successfully completed.
- Display the elapsed time on the parent's single-task correction result page.
- Remove pause and resume controls from the student timer flow.
- Reuse the existing `study_sessions` and `submissions` columns; no schema migration is required.

## Non-Goals

- Do not calculate active, focused, or foreground-only study time.
- Do not subtract time spent outside the mini-program or uploading media.
- Do not add heartbeat, WebSocket, pause-event, or activity-event tracking.
- Do not add aggregate study-time metrics to the parent home page in this change.
- Do not delete the existing pause and resume backend endpoints; they remain for compatibility but are not used by the updated client.

## Metric Definition

For a linked study session:

```text
task_elapsed_seconds = submission_completed_at - study_session.start_time
```

Both timestamps are generated by the backend. A session with no successful submission remains unfinished and is not included in completed task results.

The UI label is always `本次任务耗时`. This avoids implying that the entire interval was active study.

## Backend Design

### Starting and Recovering a Session

`POST /api/v1/study-sessions/start` remains the command for starting a task. Before creating a row, the service searches for an unfinished session for the same `daily_task_id`, defined as `end_time IS NULL` with status `running` or `paused`. If found, it returns that row instead of creating another one. A legacy paused row is treated as the task's unfinished session because pause time is not excluded by the new metric.

Add `GET /api/v1/study-sessions/active?daily_task_id=<id>`. It returns the unfinished session for that task, or `null` when none exists. A returned session contains:

```json
{
  "session_id": 123,
  "daily_task_id": 456,
  "start_time": "2026-07-19T10:00:00",
  "status": "running",
  "elapsed_seconds": 900
}
```

`elapsed_seconds` is calculated on the server from the current server time and `start_time`. This avoids client timezone parsing becoming part of the persisted metric.

### Linking and Completing a Submission

When creating a submission with `linked_study_session_id`, the backend verifies that:

- the session exists;
- the session belongs to the same `daily_task_id` as the submission;
- the session belongs to the same student as the task.

An invalid link returns HTTP 422 and does not create a submission.

When `POST /api/v1/submissions/{id}/complete` succeeds, the backend sets `submission.submitted_at` once and finishes the linked session. `finish_session` remains idempotent: an already completed session keeps its original `end_time` and `duration_seconds`.

A submission without a linked session is still allowed. Its correction result reports zero elapsed seconds and the parent UI displays `未记录`.

## Mini-Program Design

### Student Timer Page

On page show, the timer page requests the active session for its task. If one exists, it restores `sessionId`, initializes the displayed elapsed value from `elapsed_seconds`, and continues the local display tick. The local interval is visual only; it never writes the authoritative duration.

Before start, the page shows `开始`. After start or recovery, it shows a non-interactive `计时中` state and the `去上传作业` command. Pause and resume buttons are removed.

### Upload Page

The timer page continues passing `session_id` when navigating to upload. If the upload page is opened without one, it queries the active-session endpoint by task ID before creating the submission. This covers a cold restart or a route that did not preserve page state.

The upload page creates the submission only after active-session recovery completes. The linked session ID sent to the backend is therefore either the verified active session or `null`.

### Parent Task Result

The parent task-result controller formats `result.result.study_duration_seconds` for display. The correction page shows:

```text
本次任务耗时  25 分 18 秒
```

When the value is zero or missing, it shows `本次任务耗时  未记录`.

## Error Handling

- Failure to load an active session does not invent a local session ID. Starting and uploading remain available, and existing page error handling reports request failures.
- An invalid session-to-task association is rejected before submission creation.
- Repeated start requests return the same unfinished session.
- Repeated completion does not change the original elapsed duration.
- Very long elapsed intervals are preserved because they accurately represent the defined start-to-submit metric; anomaly labeling is outside this change.

## Testing

Backend tests cover:

- repeated start returns one unfinished session;
- the active-session endpoint returns server-calculated elapsed time;
- active-session recovery returns no session after completion;
- submission creation rejects a session from another task or student;
- successful completion sets `submitted_at` and finishes the linked session once;
- completion without a linked session remains valid.

Mini-program tests cover:

- timer page restores an active session when shown;
- upload page recovers the active session when no route session ID exists;
- pause and resume controls are absent;
- parent task result formats and displays task elapsed time, including the unrecorded state.

## Compatibility

The API change is additive except for stricter validation of an explicitly supplied `linked_study_session_id`. Existing database rows and clients that omit this field continue to work. No database migration or new environment variable is required.
