# 老师式整页作业批改设计

## 背景

学生提交作业后，当前结果页只显示轮询中的简单文案和拆分较细的题目列表。学生无法直观看到系统正在处理什么，也无法把错误反馈与自己实际拍摄的作业页对应起来。

本次改造把学生提交的每张照片视为一页作业。批改完成后，结果页按上传顺序完整展示这些照片，并把绿色勾、红圈、红叉和简短批语直接叠加在卷面对应位置，形成接近老师批改纸质作业的体验。

## 目标

- 提交后显示真实、可恢复的批改阶段，允许学生离开等待页。
- 上传几张照片，结果页就按顺序展示几页完整批改卷面。
- 批改结果按大题号聚合；同一大题下的小问不拆成独立结果。
- 正确结果使用绿色，错误结果使用红色。
- 错误位置直接在学生提交的原图上用红圈、红叉和批语标出。
- 保留原始照片，不把批改痕迹写入原文件。
- 定位不可靠时宁可不画标记，也不在错误位置误导学生。
- 第一版只提供小程序内状态提醒，不接入微信订阅消息。

## 非目标

- 不支持学生提交 PDF、DOCX 或其他文档格式；学生端仍提交照片或现有视频格式。
- 卷面标注只应用于照片；现有视频作业继续使用原有文字结果和家长复核流程。
- 不生成可下载或分享的合成批改图片。
- 不引入第二次视觉模型调用来专门定位题目。
- 不提供逐题点击后再跳转到红圈位置的交互。
- 不改变家长现有的“确认批改结果 / 要求重新提交”业务规则。

## 用户体验

### 提交与等待

学生完成照片上传并提交后进入批改进度页。页面显示已上传页数和以下阶段：

1. `uploaded`：上传完成。
2. `recognizing`：正在读取照片并识别题目。
3. `grading`：正在按大题批改。
4. `annotating`：正在整理卷面标注。
5. `corrected` 或 `needs_review`：批改完成。

进度页提供“先离开，稍后查看”。学生离开后，今日任务卡片继续显示“识别中”“批改中”或“生成批注中”；任务完成后显示“批改完成”。重新进入结果页时从服务器恢复当前阶段，不重新提交，也不重新开始本地计时。

阶段必须由后端任务真实更新，不用纯时间动画伪造进度。`recognizing` 对应媒体读取和请求组装，`grading` 对应视觉模型请求，`annotating` 对应模型响应规范化、坐标校验和结果落库。

### 整页批改结果

结果页顶部只展示作业名称、总页数和整体结果。正文按 `SubmissionMedia.sort_order` 纵向展示学生提交的完整照片：上传两张就展示两页，上传三张就展示三页。

每页照片上直接叠加：

- 正确答案附近的绿色勾。
- 错误答案区域的红圈和红叉。
- 必要且简短的红色批语。

每页下方只保留一行大题汇总，例如“第 1、2、4 题正确；第 3 题错误”。页面不把每道题渲染为独立卡片，也不要求学生点击题目后才查看原图标注。

无法可靠定位的题目不在照片上绘制标记。该页下方显示“第 6 题需要家长确认”等文字，状态使用中性色或现有待复核色，不误用红色或绿色。

学生端和家长端展示同一组批改卷面。家长端额外保留复核操作区。

## 后端设计

### 数据模型

在 `Submission` 增加：

- `processing_stage: str | None`：当前批改阶段。
- `processing_message: str | None`：可直接展示的简短状态说明。

在 `QuestionResult` 增加：

- `source_media_id: int | None`：该大题所属的学生提交照片。
- `annotations_json: Text | None`：规范化后的卷面标注数组。

`annotations_json` 中每项结构为：

```json
{
  "kind": "correct_tick | error_circle | error_cross | comment",
  "x": 0.25,
  "y": 0.42,
  "width": 0.18,
  "height": 0.06,
  "text": "“轻轻”更准确",
  "confidence": 0.91
}
```

坐标使用相对于原图宽高的 `0..1` 归一化值。`text` 只用于 `comment`，其他标注可以为空。第一版使用现有 SQLite 兼容升级方式为已有数据库增加列，不引入新的迁移框架。

### 视觉模型输出

视觉模型按学生照片的上传顺序接收图片，并输出：

- `source_image_index`：从 1 开始的图片序号。
- `question_no`：大题号。
- `is_correct`：正确、错误或无法判断。
- `recognized_answer`、`expected_answer` 和简短 `explanation`。
- `confidence_score`：批改置信度。
- `annotations`：该大题在原图上的标注区域。

提示词明确要求以印刷的大题号为聚合单位。`(1)(2)(3)` 等小问只能写入同一大题的说明和标注数组，不能产生多条同号或派生题号结果。

单次视觉模型调用同时返回批改和定位信息。后端用 `source_image_index` 映射真实的 `SubmissionMedia.id`，模型不接触数据库 ID。

### 结果规范化

模型响应落库前执行以下规则：

1. 以规范化后的大题号和 `source_media_id` 合并重复结果。
2. 合并小问说明，并汇总同一大题的标注数组。
3. 校验 `source_image_index` 必须对应本次提交中的作业照片。
4. 把 `x`、`y`、`width` 和 `height` 限制在 `0..1`，拒绝面积为零或明显越界的区域。
5. 标注定位置信度低于配置阈值时丢弃该标注，但保留文字批改结果。
6. `is_correct` 为未知时不生成绿色或红色标注，并使结果进入现有家长复核流程。
7. 单页标注解析失败只降级该页，不使其他页面或整份文字结果失败。

### 状态更新

Celery 批改任务在关键边界提交阶段更新：

- 接收任务并完成媒体检查后写入 `recognizing`。
- 发起视觉模型请求前写入 `grading`。
- 收到可解析响应后写入 `annotating`。
- 结果落库后写入 `corrected` 或 `needs_review`。
- 异常时沿用现有 `failed` 状态，同时保留原始照片和明确错误信息。

阶段更新应独立提交，使小程序轮询期间能够读取最新阶段。

### 结果接口

`GET /results/tasks/{task_id}` 在现有结构上增加：

- `submission.processing_stage` 和 `submission.processing_message`。
- `pages`：按 `sort_order` 排序的作业照片，包含 `media_id`、可访问图片地址、页码和该页汇总。
- 每页的 `questions`：按大题号聚合的结果。
- 每道大题的规范化 `annotations`。

接口不返回服务器本地存储路径。远程私有对象使用已有签名 URL 机制生成临时可访问地址。结果接口在返回学生照片前校验当前用户对该学生和任务的访问权限。

## 小程序设计

### 状态恢复与轮询

`result-state` 根据服务端 `processing_stage` 生成阶段标题和说明。结果页在可见时轮询，在 `onHide` 或 `onUnload` 停止定时器，在 `onShow` 立即刷新。轮询从短间隔开始并逐步降低频率，终态后停止。

今日任务卡片使用同一状态映射，确保学生离开等待页后仍能看到当前进度。第一版不请求微信订阅消息权限。

### 批改卷面组件

新增可复用的整页批改组件，供学生结果页和家长结果页共同使用。组件接收：

- 原图地址和页码。
- 该页大题结果。
- 规范化标注数组。

图片使用保持宽高比的整宽展示。标注层与图片放在同一个相对定位容器中，使用百分比坐标定位：

- `error_circle` 使用红色椭圆边框。
- `error_cross` 使用红色叉号。
- `correct_tick` 使用绿色勾号。
- `comment` 使用受边界约束的红色短批语。

标注层不改变原始图片 URL。组件在图片加载完成前不展示标注，避免因未知尺寸产生位置跳动。批语靠近边缘时向图片内部收拢，避免被裁切。

### 页面结构

学生结果页移除当前逐题卡片循环，改为：

1. 总体结果摘要。
2. 按上传顺序排列的完整批改卷面。
3. 每页一行大题汇总。
4. 回到任务计划操作。

家长结果页复用相同卷面列表，并在列表之后保留现有复核提示和操作按钮。

## 错误与降级策略

- 结果请求失败：保留当前已展示状态，提供重试，不清空原有页面。
- 图片加载失败：显示对应页码和“图片加载失败”，允许单页重试。
- 无标注但有文字结果：展示完整原图和页下汇总。
- 部分标注无效：只过滤无效项，继续展示其余标注。
- 模型无法判断：进入 `needs_review`，不绘制红绿结论标记。
- 批改任务失败：保留原始提交，允许基于同一提交重新发起批改；不要求学生重新选择照片。只有媒体本身缺失或损坏时才要求重新上传。

## 测试与验收

### 后端自动化测试

- 相同大题号的多个小问合并为一条结果。
- 不同照片上的相同题号不会跨页错误合并。
- 图片序号正确映射到 `SubmissionMedia.id`。
- 坐标归一化、越界过滤、零面积过滤和低置信度降级。
- 单页无效标注不影响其他页和文字结果。
- 阶段按照 `recognizing -> grading -> annotating -> corrected/needs_review` 推进。
- 结果接口按上传顺序返回页面、汇总和标注。
- 未授权用户不能读取学生提交照片。
- 家长确认和要求重新提交的既有测试继续通过。

### 小程序自动化测试

- `result-state` 正确映射所有处理中阶段和终态。
- 页面按 `sort_order` 排序，不按接口偶然返回顺序展示。
- 正确、错误和待复核使用不同且正确的视觉状态。
- 无标注、部分标注和图片加载失败均能降级展示。
- 页面隐藏时停止轮询，重新显示时立即恢复刷新。

### 手工验收

使用包含两张照片、八道大题且至少两道错题的提交进行验收：

- 提交后能看到真实阶段，并可安全离开等待页。
- 今日任务卡片持续显示批改状态。
- 完成后结果页完整展示两页原图，顺序与上传一致。
- 绿色勾和红色圈叉位于对应答案附近，旋转或缩放显示时不漂移。
- 每道大题只有一个结果，小问错误写在该题的简短说明中。
- 两页下方的汇总与卷面标注一致。
- 学生端和家长端看到相同卷面，家长仍可完成复核操作。

## 实施边界

改动集中在批改结果数据、批改任务阶段、结果接口、学生与家长结果页及其测试。现有上传入口、作业计划生成、学习计时和通知系统不做无关重构。
