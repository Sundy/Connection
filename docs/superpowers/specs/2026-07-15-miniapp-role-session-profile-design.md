# 小程序角色会话与“我的”页简化设计

## 背景

小程序当前会把 `token` 和 `currentRole` 写入本地缓存，但登录页始终是应用首屏，也不会读取这些状态并自动恢复会话。因此用户每次进入都需要重新选择“家长”或“学生”。“我的”页同时展示身份、统计、个人资料、孩子选择、家庭邀请码和加入家庭等多张卡片，主要操作不够集中。

## 目标

- 用户首次选择角色后，后续打开小程序自动恢复该角色并进入对应首页。
- 缓存 Token 失效时，按已保存角色静默重新登录，不要求再次选择。
- 用户只能从“我的”页主动切换角色，切换前需要确认。
- 简化“我的”页的信息层级，同时保留个人资料、孩子选择、家庭邀请和加入家庭能力。
- 保持家长与学生为两个独立登录身份，不混用两者的家庭或学生上下文。

## 非目标

- 不修改后端登录、用户、家庭或学生数据模型。
- 不合并家长与学生账号。
- 不新增独立设置页或新的底部导航项。
- 不改变现有家庭邀请码和个人资料接口契约。

## 方案选择

采用“登录页作为会话入口”的方案。登录页仍是 `app.json` 中的首个页面，但会根据本地状态呈现初始化、首次选择或重试状态，并在恢复成功后重启到目标首页。

不在 `App.onLaunch` 中直接跳转，因为异步读取缓存和页面初始化可能产生跳转竞态。也不只依赖缓存 Token，因为 Token 过期后仍应按保存角色自动重新登录。

## 会话状态设计

新增独立、可单测的会话状态工具，统一提供以下纯逻辑：

- 校验保存的角色是否为 `parent` 或 `student`。
- 根据角色返回家长首页或学生首页。
- 判断当前是否属于首次选择场景。
- 生成登录成功后需要持久化的会话状态。

应用启动流程：

1. 登录页同步读取 `currentRole` 和 `token`，避免依赖 `App.onLaunch` 的异步回调顺序。
2. 没有合法保存角色时，显示首次角色选择按钮。
3. 有角色和 Token 时，将 Token 写入全局状态并调用 `auth.me()` 恢复用户上下文。
4. 上下文有效且用户角色与保存角色一致时，写入家庭、学生等全局状态并进入对应首页。
5. Token 缺失、失效或返回角色不一致时，调用 `auth.login(savedRole)` 静默重新登录，再加载上下文并进入首页。
6. 网络或服务异常时保留角色偏好，显示失败说明和“重试”按钮，不重新显示角色选择。

成功导航使用 `wx.reLaunch`，清除登录页及旧角色页面栈，避免返回到错误身份的页面。

## 角色切换设计

“我的”页顶部身份区域提供“切换身份”入口。点击后弹出确认框，明确将从当前角色切换到目标角色。

- 取消确认：不修改任何缓存或全局状态。
- 确认切换：调用 `auth.login(targetRole)`，成功加载目标身份上下文后，原子地更新 `token`、`currentRole`、当前家庭和当前学生，并 `reLaunch` 到目标首页。
- 切换失败：保留原身份和缓存，停留在“我的”页并显示错误提示。

现有 `auth.localOpenid(role)` 已按角色使用不同的本地 OpenID，因此家长和学生会继续对应独立账号。只有目标身份完整登录并加载上下文成功后，才提交新的角色与 Token，防止半完成状态。

## “我的”页信息架构

页面采用分组列表布局，延续现有绿色视觉体系，但减少卡片数量、阴影、说明文字和纵向留白。

### 身份摘要

顶部保留一个强调区，展示当前角色和家庭名称。右侧提供“切换身份”操作。移除家长数、孩子档案数和学生账号数统计。

### 个人资料

默认只显示昵称摘要和首字头像。点击该行后在当前页面展开编辑区域：

- 家长编辑昵称。
- 学生编辑昵称、年级和学校。
- 保留随机昵称能力。
- 保存成功后刷新上下文并自动收起编辑区。

### 家长分组

- “当前孩子”行展示当前孩子姓名和年级，点击后展开现有孩子列表并允许选择。
- “家庭邀请码”行展示邀请用途，进入展开态后显示邀请码和复制按钮。
- 没有孩子时显示简短空状态，不恢复已移除的“添加孩子”表单。

### 学生分组

- 已加入家庭时展示绑定学生档案摘要，不显示家庭管理入口。
- 未加入家庭时显示“加入家庭”行，点击后展开邀请码输入框和提交按钮。

底部角色导航保持不变。

## 状态与错误处理

- 登录页包含 `initializing`、`selecting`、`error` 三类视图状态，防止自动恢复期间短暂闪出角色按钮。
- 自动恢复和角色切换都要防止重复点击。
- 本地存储写入失败会记录警告；当前运行时仍使用已获得的会话，不因单次缓存失败阻断登录。
- `auth.me()` 的鉴权失败和网络失败需要区分：鉴权失败触发静默重新登录，网络失败直接进入重试状态，避免无意义地重复调用微信登录。
- 如果现有请求层无法可靠识别鉴权失败，则补充标准化的状态码信息，但不改变业务接口返回格式。

## 测试策略

遵循现有 Node `node:test` 测试模式，并在实现前先添加失败测试。

- 会话工具测试：角色校验、首次选择判断、目标首页和会话更新数据。
- 登录页行为测试：存在保存角色时不显示选择按钮；初始化、失败和重试结构完整；成功导航使用 `reLaunch`。
- 切换行为测试：取消不改变状态；成功后更新角色与 Token；失败保留原身份。
- “我的”页结构测试：统计已移除，个人资料和家庭功能使用可展开分组，切换入口存在。
- 可见性测试：家长、已绑定学生、未绑定学生只看到各自适用的功能。
- 回归验证：运行 `node --test miniapp/tests/*.test.js` 和相关后端登录/家庭流程测试。

## 验收标准

- 新用户首次打开时需要选择一次角色。
- 已选择角色的用户关闭并重新打开小程序后，无需再次选择即可进入对应首页。
- 缓存 Token 无效时仍能按已保存角色自动恢复。
- 用户可在“我的”页确认后切换身份，取消或失败不会改变当前身份。
- “我的”页默认态只显示身份摘要和必要的分组行，编辑表单与详细列表按需展开。
- 家长和学生的数据上下文不会在自动恢复或切换过程中相互混用。
