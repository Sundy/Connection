# 学习计划与真实批改改造设计

## 目标

让家长端和学生端能够清楚地按日期、科目查看学习计划，移除学生提交标准答案的入口，并把照片、视频作业接入可观测、可失败、可复核的真实 AI 批改流程。

## 范围

本次改造覆盖家长端周期计划、学生端每日任务、学生作业提交、学生批改结果、家长批改结果及其后端接口和服务。保留现有计划和任务数据模型，不增加教师端、人工在线批改工作台或消息推送，不修改现有 AI 密钥、模型和服务商配置。

## 方案选择

采用增量改造：保留现有 `AssignmentBatch`、`DailyTask`、`Submission`、`CorrectionResult` 和 `QuestionResult`，扩展接口返回值、提交状态和媒体处理流程。

没有选择重写小程序页面或重新设计计划模型，因为现有数据已经包含 `task_date` 和 `subject`，问题主要是查询能力、页面组织方式以及批改失败被模拟结果掩盖。增量方案能兼容已有记录并降低迁移风险。

## 日期与科目计划

### 学生端

学生每日任务页显示语义化日期，例如“今天 · 7月11日”“昨天 · 7月10日”或“7月15日 · 星期三”。顶部提供前一天、回到今天、后一天以及系统日期选择器。

页面以科目为一级分组，每组展示该科目的已完成数量和总任务数。顶部提供“全部”和当前日期实际存在的科目筛选项。切换日期或科目不会改变任务数据，只改变当前查询和显示范围。

空状态必须区分“今天没有任务”和“所选日期没有任务”。从结果页返回时保留合理的日期上下文；无法保留时回到今天。

### 家长端

周期计划按日期分组，日期内再按科目分组。顶部提供计划日期选择器和前后日期切换，只允许在计划开始、结束日期范围内浏览。每个日期显示任务总数与完成数，每个科目显示科目任务进度。

已有任务详情、原文件预览和批改结果入口保持可用。

### 后端

`GET /tasks/today` 保留名称和默认当天行为，正式支持 `target_date=YYYY-MM-DD`。返回当前日期、语义展示所需的基础字段、任务列表和按科目汇总。

计划日历接口继续返回扁平 `items` 以兼容旧客户端，同时返回计划起止日期和日期、科目汇总。科目分组可以由小程序使用扁平任务确定性生成，不在数据库保存重复的分组结构。

## 作业提交

学生端只保留拍照/相册和拍视频。删除答案文本、答案图片、答案文件、答案状态提示以及相关事件和请求参数。

后端保留历史 `answer_text` 字段和 `SubmissionMedia.purpose == "answer"` 数据，不做破坏性迁移。新的学生提交请求模型不接受 `answer_text`，提交更新接口不再用于学生补充答案。批改所需标准答案仅来自家长导入的 `AssignmentItem.answer_text` 或家长提供的原始资料。

上传的作业媒体必须至少有一个 `purpose=homework` 项才允许完成提交。不存在任务、提交或媒体时，接口返回明确的 404 或 422 错误，而不是触发空对象异常。

## 批改状态机

提交状态使用以下状态：

- `draft`：已创建但还没有可批改媒体。
- `uploaded`：至少一个作业媒体上传成功。
- `processing`：批改任务已经开始。
- `corrected`：真实 AI 结果已保存，且无需人工复核。
- `needs_review`：结果已保存，但模型低置信度、任务类型不明确或媒体不足以形成可靠判断。
- `failed`：AI、媒体处理或结果解析失败，没有生成可信批改结果。

每日任务状态与最终提交状态同步。批改失败必须保存经过脱敏的错误类别和用户可读消息；不得记录 API Key、认证头、图片二进制或完整 AI 请求内容。

系统不得以固定分数、固定题号或固定识别答案回退。AI 未配置或调用失败时进入 `failed`。AI 返回内容不完整但仍具有可用结论时进入 `needs_review`；完全不可解析时进入 `failed`。

## 照片批改

照片批改输入包括任务标题、科目、任务要求、家长导入的原始资料、可用标准答案和学生作业图片。视觉模型返回结构化总评和逐题结果。

服务端对模型结果进行结构校验和归一化：分数限制在 0 到 100，置信度统一到 0 到 1，逐题结果允许 `is_correct` 为空表示无法可靠判断。低于配置阈值或存在无法判断题目时，提交进入 `needs_review`。

有效结果写入 `CorrectionResult` 和 `QuestionResult`。不得因为模型漏掉可选字段而伪造题目答案。

## 视频批改

系统依据任务的 `task_type`、`submit_type`、标题和要求选择处理方式：

- 朗读、背诵、口语类使用音频提取和 ASR 转写，将转写文本、任务要求和参考资料交给模型评估。
- 书写、计算过程、操作展示类按受控间隔抽取有限数量关键帧，将关键帧和任务要求交给视觉模型评估。
- 无法可靠分类时可以同时使用转写和关键帧，但最终必须标记 `needs_review`。

视频时长和抽帧数量遵循现有配置上限，避免无限制地向模型发送内容。音频提取、转写、抽帧或视觉调用失败时进入 `failed`；只有部分信息可用时可以生成结果，但必须进入 `needs_review`。

## 结果展示

结果接口保留原有 `task`、`submission`、`result` 和 `questions`，并增加提交状态、用户可读错误、复核原因和逐题置信度。

学生结果页根据状态展示：

- `processing`：显示批改中并继续有限时长轮询。
- `corrected`：显示总评、分数和逐题结果。
- `needs_review`：显示已有结果及“需要家长复核”，不把不确定项表现为确定错误。
- `failed`：停止轮询，显示失败原因和重新提交入口。

轮询达到上限后显示“处理时间较长，可稍后回来查看”，避免无限轮询。网络错误应显示可重试提示，不得被空 `catch` 静默忽略。

家长结果页展示相同状态，并额外展示复核原因、逐题识别答案、参考答案、解释和置信度。

## 异步执行与可靠性

保留 Celery 任务和本地 eager 模式。提交完成接口只负责校验、更新状态和调度任务。批改任务自身负责把状态切换为 `processing`，捕获已知异常，写入 `failed`，并保证数据库会话关闭。

重复调度同一个已完成提交时不得重复创建批改结果。重新提交产生新的 `Submission`，结果接口继续选择该任务最新一次提交及与该提交关联的结果，避免展示旧提交结果。

## 数据兼容

现有 SQLite 开发数据库通过当前兼容式启动迁移补充错误信息等新增列，不删除历史列。部署数据库仍需用等价的显式迁移执行相同字段变更。

已有包含学生答案的提交记录继续可读，但新客户端不展示或写入这些数据。旧客户端发送答案字段时，接口应按照明确的请求校验策略拒绝，防止新记录继续产生学生答案。

## 测试与验收

后端自动化测试覆盖：

- 默认当天和指定日期查询。
- 科目汇总与计划日期范围。
- 学生提交模型拒绝答案字段。
- 没有作业媒体时拒绝完成提交。
- 照片结构化批改结果落库。
- 朗读视频走转写分支。
- 书写视频走抽帧分支。
- 不明确视频进入复核状态。
- AI 调用、媒体处理和 JSON 校验异常进入失败状态。
- 低置信度进入复核状态。
- 重复任务不会产生重复结果。
- 最新提交不会错误读取旧结果。
- 代码中不再生成固定模拟分数和固定题目。

小程序验收包括 JavaScript 语法检查，以及在微信开发者工具中手动验证日期切换、科目筛选、家长计划分组、照片/视频提交、四种结果状态和重新提交入口。

## 实施顺序

1. 学生端日期切换和日期语义展示。
2. 家长端、学生端科目分组及进度。
3. 移除学生端答案上传和后端新写入能力。
4. 建立真实批改状态、错误落库和结果接口。
5. 接通照片逐题批改。
6. 接通视频转写与抽帧分支。
7. 完善学生端和家长端结果展示、复核与失败重试。
8. 执行全量后端测试和小程序手动验收。
