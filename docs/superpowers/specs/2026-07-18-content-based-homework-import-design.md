# 内容命名、增量作业与答案匹配设计

## 背景与根因

家长上传图片时，微信客户端经常只提供 `tmp_...` 临时路径。现有导入接口把这个临时名称保存为 `ImportFile.file_name`，计划生成又用文件名去掉扩展名作为 `AssignmentItem.title`，因此确认计划页出现长串临时文件名。

现有 `generate_plan_from_import()` 在同一导入计划仍处于待确认状态时，会先删除该计划的所有 `DailyTask` 和 `AssignmentItem` 再重新生成。这使再次上传或再次生成表现为替换，而不是增加。

当前 `ImportFile.file_type` 只表示图片、PDF、Word 等媒体格式，没有“作业”或“答案”的文档角色。系统也没有答案文件与作业文件的匹配关系、匹配置信度或确认拦截机制，无法保证标准答案属于对应作业。

## 目标

- 作业名称必须根据文件内容生成中文语义名称，不再使用临时文件名或 UUID 作为标题。
- 同一孩子、相同日期范围多次上传时，新作业只追加，不替换已有计划内容。
- 上传文件和本次新增、尚未生效的作业项可以删除。
- 答案文件可选；一份答案只能匹配一份当前批次作业文件。
- 上传了答案但没有高置信度匹配时，保留答案文件并禁止确认计划。
- 已生效的历史作业、任务、提交和批改结果不通过本入口删除或修改。

## 非目标

- 不允许家长手动修改 AI 生成的作业名称。
- 不允许手动强制建立低置信度答案匹配。
- 不支持一份答案对应多份作业，也不支持一份作业对应多份答案。
- 不允许后来给已经生效或已经批改的历史作业补换答案。
- 不改变现有多天日期范围内的作业分配算法。

## 方案选择

采用“文件级暂存与一对一匹配”方案。每次上传建立独立导入批次，批次只表示本次新增内容。作业和答案逐份解析、命名和匹配，确认后再把本次新增内容并入对应计划。

没有采用“上传后立即写入生效计划”，因为异步识别失败会在正式计划中留下半成品，删除和回滚也会影响已有任务。没有采用“合并全部文件后统一拆分”，因为它会丢失作业与答案的文件级一对一关系，容易把答案串到错误作业。

## 数据模型

### ImportFile

保留 `file_name` 作为物理文件原名，仅用于存储、预览、下载和排查。新增：

- `document_role`：`homework` 或 `answer`。
- `recognized_title`：根据内容生成的中文作业名；答案可使用对应作业名派生展示名。
- `recognition_status`：`pending`、`processing`、`success`、`failed`。
- `recognition_error`：不包含凭据和完整文件内容的用户可读失败原因。
- `content_summary`：用于匹配的受控长度内容摘要。
- `content_signature_json`：结构化内容特征，包括学科、年级线索、教材/章节、题号范围、题目数量和关键题干指纹。
- `match_status`：作业为 `not_required`；答案使用 `pending`、`matched`、`unmatched`。
- `matched_homework_file_id`：答案匹配的唯一作业文件 ID，可空、自关联外键。
- `match_confidence`：0 到 1。
- `match_reason`：匹配或不匹配的简短原因。

数据库对非空 `matched_homework_file_id` 建唯一约束，使一个作业最多被一份答案占用。服务层同时验证答案和作业属于同一导入批次、角色正确且尚未确认。

### AssignmentBatch

每个导入批次先生成一个独立的待确认 `AssignmentBatch`。新增可空 `target_assignment_batch_id`，指向同一孩子、相同 `period_type`、`start_date` 和 `end_date` 的已有生效计划。

- 没有目标计划时，确认后当前暂存计划直接变为 `active`。
- 存在目标计划时，确认时把本次暂存的 `AssignmentItem` 和 `DailyTask` 移入目标计划，累加预计时长，并把暂存计划标记为 `merged`。

目标计划只考虑日期范围完全相同的计划，避免把不同周期的计划误合并。多天范围继续按现有算法分配；单日范围的新增任务全部进入该日期。

`target_assignment_batch_id` 是生成确认页时的目标提示，不作为最终唯一判断。确认事务会再次查询并锁定同一孩子、相同日期范围的生效计划；若另一个暂存计划已经先确认生效，当前新增内容改为并入该计划，避免两个连续确认产生两个互相不可见的当日计划。

### AssignmentItem

一个 `document_role=homework` 文件最多生成一个作业项，`import_file_id` 用于幂等识别。标题始终复制自 `ImportFile.recognized_title`。

匹配成功时，把答案文件的解析正文写入 `AssignmentItem.answer_text`。匹配答案文件自身仍保存在 `ImportFile`，用于预览、追踪和删除级联。

## 上传与识别流程

上传页提供两个明确入口：

- “上传作业”上传 `document_role=homework`。
- “上传答案（可选）”上传 `document_role=answer`。

上传接口继续保存真实原名或微信临时名到 `file_name`，但 API 和界面不能把它作为作业标题。识别完成前，界面显示“正在识别第 N 份作业”或“正在识别第 N 份答案”。

文件上传成功后进入异步处理：

1. 图片走 OCR，PDF、Word、Excel 等走对应文档解析。
2. 解析服务保存完整 `extracted_text`。
3. 内容识别服务输出结构化签名和推荐中文标题。
4. 作业文件更新名称和特征后，触发本批次答案重新匹配。
5. 答案文件更新特征后，立即与本批次已识别作业匹配；若作业尚未完成识别，保持 `pending`。

重复调用解析只处理 `pending`、`failed` 或新上传文件，不重复处理已经成功且内容未变化的文件。

## 内容生成中文作业名

内容识别模型输出：

- 学科。
- 年级线索。
- 教材、单元或章节。
- 练习类型。
- 题号范围与题目数量。
- 推荐标题。
- 置信度。

服务端对标题执行确定性清理：

- 去掉文件扩展名、`tmp`、UUID、时间戳和连续无意义字符。
- 去掉重复的“作业”“练习”等尾缀。
- 保留内容中可以证实的学科、年级、章节和练习类型。
- 限制为适合小程序卡片展示的长度，不截断关键章节或题号信息。

示例：

- “数学四年级下册第 3 单元练习”
- “语文阅读理解专项练习”
- “英语 Unit 4 单词默写”

不使用文件名作为标题回退。若内容不足以生成可靠名称，文件显示“作业内容无法识别”，确认计划被阻止。由于产品不提供手动改名，用户只能删除后重新上传清晰文件。

## 答案一对一匹配

答案只在当前导入批次内匹配本次新增作业，不搜索或修改已生效历史作业。

匹配特征包括：

- 学科、年级、教材、单元和章节一致性。
- 主问题题号范围及三级小题结构一致性。
- 题干关键词、算式或阅读材料指纹重合度。
- 答案数量与作业题目结构的一致性。
- 文档是否包含答案、解析、评分标准等答案特征。

匹配服务对全部可用作业—答案组合评分，然后执行一对一分配。只有达到高置信度阈值且不存在近似冲突的组合才标记 `matched`。

低置信度、范围冲突或无法判断时标记 `unmatched`，并提供原因，例如：“答案包含第 1–20 题，但作业为第 21–40 题”。上传了答案而存在 `pending` 或 `unmatched` 状态时，确认计划返回冲突错误。

答案是可选项。没有上传答案的作业允许确认，后续按现有 AI 判断流程批改。上传了答案则必须成功匹配；不提供手动强制匹配入口。

## 增量生成与计划合并

`generate_plan_from_import()` 不再删除已有计划的全部作业项和每日任务。

对当前暂存计划：

- 按 `ImportFile.id` 查找已有 `AssignmentItem.import_file_id`。
- 仅为尚未生成的 `homework` 文件创建作业项和预览任务。
- 已成功生成的文件重复调用时直接复用，保证幂等。
- 答案文件不独立创建作业项，只把匹配正文赋给对应作业项。

确认时：

- 所有文件必须处理完成。
- 每个作业必须有可靠中文名称。
- 每个已上传答案必须处于 `matched`。
- 不得存在一对多或多对一占用。

若没有目标计划，当前暂存计划生效。若有目标计划，只移动本次暂存内容并合并预计时长；目标计划已有作业项、每日任务、提交和批改结果保持不变。

确认页把“已有作业”和“本次新增”分区展示。删除按钮只出现在“本次新增”区域。

## 删除行为

上传页的所有未确认文件均可删除。

- 删除答案：删除答案文件和匹配关系，作业保留并恢复为无标准答案状态。
- 删除未匹配作业：删除该作业文件。
- 删除已匹配作业：先弹窗说明配对答案也将删除；确认后同时删除作业和答案。
- 在确认计划页删除本次新增作业项：删除作业项、对应暂存每日任务、作业文件和配对答案。

删除只允许发生在导入批次和暂存计划尚未确认时。已经生效的历史作业、任务、提交和批改结果不可通过这些接口删除。

服务端先验证家庭归属和暂存状态，再删除数据库关系、本地缓存和 OSS 对象。若存储删除失败，接口返回明确错误，客户端保留卡片并允许重试，不能显示虚假的成功状态。

## API 与小程序

上传接口增加 `document_role` 表单字段。文件列表和上传响应增加：

- `display_name`：识别成功后使用中文作业名；识别中使用序号占位文案；不返回临时文件名作为主要展示。
- `document_role`
- `recognition_status`
- `match_status`
- `matched_homework_file_id`
- `match_confidence`
- `match_reason`
- `can_delete`

新增删除上传文件接口和删除暂存作业项接口。所有导入、文件、解析、计划生成和确认接口统一校验当前用户是否属于该孩子的家庭。

上传页分别显示作业和答案列表，每张卡片提供删除操作和处理状态。答案匹配成功时显示“《作业中文名》答案”，未匹配时显示原因。页面重新进入时通过文件列表接口恢复当前批次，不依赖内存中的临时数组。

确认页显示：

- 已有作业：只读。
- 本次新增：可删除，显示中文作业名、学科、日期安排、答案匹配状态和原文件预览入口。

## 错误处理

以下状态禁止确认并返回可操作原因：

- 文件仍在上传、解析、内容识别或答案匹配中。
- 作业内容不足以生成可靠中文名称。
- 答案处于等待匹配或未匹配状态。
- 作业或答案发生一对一关系冲突。
- 删除存储对象失败。

AI/OCR 请求失败时保留已上传文件并标记失败，允许重新解析或删除重传。错误信息不得包含 API Key、认证头、文件二进制或完整敏感正文。

## 权限与安全

创建导入批次时验证学生属于当前家长家庭。上传、列表、预览、解析、删除、计划生成和确认均从批次或计划反查学生和家庭，拒绝跨家庭访问。

删除接口只接受数据库解析出的具体文件 ID，不接受客户端提供的任意文件路径或 OSS Key。物理删除目标必须来源于该 `ImportFile` 的受控存储字段。

## 数据迁移与兼容

MySQL 幂等增加上述可空字段和约束。历史 `ImportFile` 的 `document_role` 可保持为空或在读取时兼容为 `homework`，不批量重新识别历史文件。

历史 `AssignmentItem.title`、任务、提交和批改结果保持原值。新逻辑只影响新导入批次。旧客户端未提供 `document_role` 时，上传接口兼容为 `homework`，但新小程序始终显式传值。

## 测试与验收

后端测试覆盖：

- 微信临时文件名不会进入新作业标题或主要 API 展示名。
- 数学、语文、英语内容生成合理中文标题。
- 内容无法识别时阻止确认。
- 答案正确匹配一份作业并写入 `answer_text`。
- 答案题号、章节或学科冲突时保持未匹配并阻止确认。
- 一份答案不能匹配多份作业，一份作业不能占用多份答案。
- 答案先上传时等待，作业完成识别后自动重试。
- 同一孩子、相同日期范围重复上传只追加新作业。
- 重复生成同一批次不重复创建作业项或每日任务。
- 合并时旧作业、旧任务、提交和批改结果不被删除或修改。
- 删除答案保留作业；删除已匹配作业级联删除答案。
- 删除暂存作业项同步删除其预览任务和文件。
- 已生效计划拒绝通过暂存删除接口删除。
- 跨家庭上传、查看、解析、删除和确认均被拒绝。
- 存储删除失败不会返回成功。

小程序测试覆盖两个上传入口、内容名称展示、识别和匹配状态、删除确认、未匹配答案禁用确认、已有/新增分区，以及重复进入页面后的文件列表恢复。

手工验收使用至少两份不同数学作业和一份对应答案：验证答案只匹配正确作业；再次为同一孩子和日期范围上传第三份作业，确认旧两份仍存在且新作业追加到对应日期。
